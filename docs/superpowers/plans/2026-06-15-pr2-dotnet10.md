# PR 2 — .NET 10 Upgrade Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move the FHIR Server build from `net9.0;net8.0` to `net10.0;net8.0` — SDK, target frameworks, framework-tied NuGet packages, the shared `Microsoft.Health.*` line (10 → 11), CI pipelines, the production Docker image, and the dev container — with no functional change.

**Architecture:** Pure framework/tooling bump. Lands **after PR 1 (MediatR → Medino) is merged**, so this diff carries no business-logic churn. `net8.0` stays as the downlevel floor (the fhir-paas RP consumes the net8.0 asset via downlevel), so every change is "add net10, keep net8."

**Tech Stack:** .NET SDK 10.0.300, `net10.0`/`net8.0` TFMs, `Microsoft.Extensions.*` / `System.*` 10.0.x, `Microsoft.Health.*` 11.0.108, Azure DevOps pipelines, Docker (azurelinux3.0), VS Code dev container.

**Source of truth:** `docs/superpowers/specs/2026-06-15-dotnet10-upgrade-design.md` §5. ADO Story #195645 (add a PR 2 task).

> **Version placeholders:** the spec deliberately leaves the exact 10.0.x patch numbers (`10.0.x`) to be pinned at implementation time. Task 1 below is a concrete discover-then-pin procedure that resolves every `10.0.x` to a real version before any file is edited. Do not leave a literal `10.0.x` in any committed file.

## Task 1: Discover and pin the concrete 10.0.x versions

**Files:** none (records values used by later tasks). Do this first so no `10.0.x` placeholder ever lands in a file.

- [ ] **Step 1: Pin the SDK version**

Use `10.0.300` (the dicom-server precedent baseline). Confirm it (or a newer stable 10.0.3xx) is installable:
```powershell
dotnet sdk check
```
Record the chosen SDK version → call it `$SDK` (e.g. `10.0.300`).

- [ ] **Step 2: Pin the runtime/ASP.NET package patch (`DotNetSdkPackageVersion` and `AspNetPackageVersion`)**

Find the latest stable `10.0.x` for the runtime libraries:
```powershell
dotnet package search Microsoft.AspNetCore.Mvc.NewtonsoftJson --source https://api.nuget.org/v3/index.json --prerelease:$false | Select-String "10.0"
dotnet package search Microsoft.Extensions.Configuration --source https://api.nuget.org/v3/index.json | Select-String "10.0"
```
Record the latest common stable `10.0.x` → call it `$RT` (e.g. `10.0.0`). Both `DotNetSdkPackageVersion` and `AspNetPackageVersion` will be set to `$RT`.

- [ ] **Step 3: Confirm the azurelinux3.0 container tags exist (Dockerfile risk)**

Verify both image tags are published before editing the Dockerfile:
```powershell
docker manifest inspect mcr.microsoft.com/dotnet/sdk:$SDK-azurelinux3.0      | Select-String "mediaType" | Select-Object -First 1
docker manifest inspect mcr.microsoft.com/dotnet/aspnet:$RT-azurelinux3.0    | Select-String "mediaType" | Select-Object -First 1
```
Expected: both return manifest JSON (no `manifest unknown`). If the exact `$RT` aspnet azurelinux tag is missing, pick the nearest published `10.0.x-azurelinux3.0` aspnet tag and record it → `$ASPNET_TAG`; record the sdk tag → `$SDK_TAG` (`$SDK-azurelinux3.0`).

- [ ] **Step 4: Record the values for the rest of the plan**

Write the four resolved values somewhere visible (PR description / scratch note):
`$SDK`, `$RT`, `$SDK_TAG`, `$ASPNET_TAG`. Every `10.0.x` below refers to one of these.
